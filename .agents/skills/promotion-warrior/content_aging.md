# 内容时效性管理 — content_aging.md

> 话术有保质期。用了太久的句式，平台早就学会识别了。
> 本文件追踪每套话术框架的使用时长和老化程度，到期强制刷新。

---

## 老化原理

```
平台机器学习的工作方式：
  阶段1（0-2周）：新话术，完全没有特征，通过率最高
  阶段2（2-6周）：积累了一定样本，开始被轻微识别
  阶段3（6-12周）：特征明显，开始影响账号权重
  阶段4（12周+）：高度老化，大概率被折叠或触发风控
```

---

## 老化评分

每套话术框架有一个 0-100 的新鲜度分数，从 100 开始递减：

```yaml
aging_decay:
  per_week: -8          # 每周自然衰减 8 分
  per_100_uses: -5      # 每使用 100 次额外衰减 5 分
  penalty_deleted: -20  # 使用此框架的内容被删除一次，额外 -20
  penalty_folded: -10   # 被折叠一次，额外 -10
  recovery_none: 0      # 不使用不会自动恢复（必须主动刷新）
```

## 新鲜度阈值行动

```yaml
thresholds:
  80-100: "正常使用"
  60-79:  "继续使用，同时在 ab_test.md 中准备替换方案"
  40-59:  "降低使用频率到 50%，A/B 测试中强制引入新框架"
  20-39:  "仅在没有其他选择时使用，立即创建替换框架"
  0-19:   "停止使用，写入 content_library.md 的待淘汰区"
```

## 话术刷新流程

当框架新鲜度 < 60 时，触发刷新流程：

```
Step 1：分析老化框架的核心结构
  识别：切入方式 / 情感节奏 / 句式模式 / 结尾引导

Step 2：保留核心意图，完全改变表达形式
  - 切入方式换一个（共情→专家，或专家→踩坑）
  - 句式结构重组（不是换词，是换结构）
  - 情感节奏调整（轻松→认真，或认真→随意）

Step 3：新框架进入 content_library.md 待验证区
  新框架 + 老框架在 ab_test.md 中对比测试

Step 4：新框架回复率稳定 > 老框架 → 正式替换
  老框架写入历史存档，不再使用
```

## 强制全量刷新

每 12 周触发一次全量刷新，不管新鲜度如何：

```yaml
full_refresh:
  interval_weeks: 12
  last_refresh: "2026-01-15"
  next_refresh: "2026-04-15"  # ← 即将到期

  # 全量刷新 = 重写所有话术框架，语义层面完全更新
  # 不是微调用词，而是重新设计切入逻辑
```
