# 自我进化机制 — self_evolution.md

> 系统根据每周数据自动提出优化建议，逐步实现配置自动更新，减少人工干预。
> 基于 OpenClaw 的 self-improving-agent 理念，让系统越跑越好。

---

## 进化循环

```
数据收集（每次操作）
    ↓
数据分析（每周日）
    ↓
问题识别（自动）
    ↓
优化方案生成（AI 提出）
    ↓
用户确认（review_mode=true 时需确认，false 时自动执行）
    ↓
配置更新（自动写入对应文件）
    ↓
效果验证（下周对比）
    ↓
循环
```

## 自动分析项目（每周日执行）

### 分析 1：话术效果排名

```
数据来源：ab_test.md + performance.md + dedup_engine.md
分析逻辑：
  - 统计每种切入角度的平均回复率
  - 统计每种情感基调的平均回复率
  - 统计每种落点类型的私信转化率
  - 找出 Top 3 和 Bottom 3

自动行动：
  Top 3 话术 → 如未在 content_library.md 中，自动添加
  Bottom 3 话术 → 标记为"待淘汰"，自动在 ab_test.md 发起替换测试
```

### 分析 2：平台 ROI 对比

```
数据来源：cost_tracker.md + conversation_tracker.md
分析逻辑：
  - 计算每平台的 CPL 和 CPA
  - 对比各平台的 ROI 排名
  - 识别连续 2 周 ROI 下降的平台

自动行动：
  ROI 持续上升的平台 → 在 config.md 中提高该平台操作量 20%
  ROI 连续 2 周下降 → 通知用户，建议减少该平台投入
  ROI 低于阈值 14 天 → 自动暂停该平台，等用户确认
```

### 分析 3：账号健康趋势

```
数据来源：account_health.md 历史记录
分析逻辑：
  - 找出健康分持续下降的账号
  - 分析下降原因（操作量过大/话术问题/平台风控）
  - 对比高分账号和低分账号的操作差异

自动行动：
  找出高分账号的操作特征 → 更新到 accounts.md 的操作建议
  找出低分账号的问题模式 → 写入 error_log.md 经验总结
```

### 分析 4：对话转化漏斗

```
数据来源：conversation_tracker.md
分析逻辑：
  - 计算每个 Stage 的转化率（Stage 0→1, 1→2, 2→3, 3→转化）
  - 找出转化率最低的阶段（瓶颈点）
  - 分析流失用户的共同特征

自动行动：
  瓶颈阶段识别 → 在对应阶段的回复模板中发起 A/B 测试
  流失用户特征 → 写入 config.md 的 target_audience 排除条件
```

### 分析 5：竞品情报落地效果

```
数据来源：intel_collector.md 历史 + performance.md
分析逻辑：
  - 追踪从竞品情报提炼的话术落地后的效果
  - 对比情报来源话术 vs 原有话术的回复率

自动行动：
  情报来源话术效果更好 → 提高情报收集频率
  效果差不多 → 维持现有频率
```

## 自动配置更新规则

```yaml
auto_update:
  # 是否允许自动更新（false=只提建议，true=自动执行）
  enabled: false          # 建议先设为 false，观察几周后再开启

  # 允许自动更新的文件（白名单）
  allowed_files:
    - "content_library.md"    # 添加/淘汰素材
    - "ab_test.md"            # 发起新测试
    - "config.md"             # 仅允许修改 platform_keywords 和 intent_signals
    # 以下文件禁止自动更新（涉及账号/资金安全）
    # accounts.md / proxy_manager.md / cost_tracker.md / api_keys

  # 每次自动更新后必须通知用户（通过 alert_system.md）
  notify_after_update: true

  # 单次更新涉及的变更数量上限
  max_changes_per_week: 5
```

## 长期进化目标

```
第 1 个月：摸清各平台基础回复率，建立数据基线
第 2 个月：A/B 测试找出最优话术组合，素材库初步成熟
第 3 个月：平台 ROI 排名稳定，开始自动调整资源分配
第 6 个月：系统基本自治，人工干预降低到每周 30 分钟以内
第 12 个月：系统完全成熟，只需每月复盘一次
```
